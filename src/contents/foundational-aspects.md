# 4. 기본적인 개념

본 섹션은 프론트엔드 웹 개발이 진행되는 환경의 기본적인 측면을 식별하고 정의합니다.

## 4.1 월드 와이드 웹 (WWW 또는 웹)

보통 웹으로 잘 알려진 월드 와이드 웹(World Wide Web)은 상호 연결된 하이퍼텍스트 문서와 리소스의 체계입니다. 인터넷을 통해 접근하며, 웹 페이지를 렌더링하기 위해 브라우저를 사용하고 사용자들이 방대한 정보와 멀티미디어를 보고, 탐색하고, 상호작용할 수 있게 합니다. 1989년 팀 버너스 리(Tim Berners-Lee)에 의해 시작된 웹은 정보 공유와 커뮤니케이션에 혁명을 일으키고 현대 디지털 시대의 근본을 마련했습니다.

더 학습하기:

- [웹의 동작 방식](https://developer.mozilla.org/ko/docs/Learn_web_development/Getting_started/Web_standards/How_the_web_works) on MDN

## 4.2 인터넷

인터넷은 전 세계에 걸쳐 연결된 컴퓨터들의 거대한 네트워크입니다. 월드 와이드 웹과 이메일, 파일 공유와 같은 서비스를 가능하게 하는 인프라입니다. 인터넷은 다양한 프로토콜과 이 방대한 네트워크에서 데이터 라우팅 역할을 하는 가장 기본적인 인터넷 프로토콜(IP)을 작동합니다.

더 학습하기:

- [Internet Fundamentals](http://internetfundamentals.com/) from Frontend Masters
- [인터넷은 어떻게 동작하는가](https://developer.mozilla.org/ko/docs/Learn_web_development/Howto/Web_mechanics/How_does_the_Internet_work) on MDN

## 4.3 IP(인터넷 프로토콜) 주소

IP 주소는 인터넷의 디바이스를 식별하는 고유한 식별자로, 물리적 세계에서 우편 번호가 위치를 식별하는 방법과 비슷합ㅂ니다. 인터넷을 통해 데이터가 정확하게 라우팅되고 전달되기 위해 IP 주소는 필수적입니다. 컴퓨터부터 스마트폰까지 인터넷에 연결된 각 디바이스는 IP 주소를 할당받습니다.

주요 IP 주소 표준는 다음의 두가지가 있습니다:

- IPv4(Internet Protocol version 4): 이는 더 오래되고 가장 널리 사용되고 있는 표준입니다. IPv4 주소는 32 비트의 길이로, 이론적으로 최대 약약 43억 개의 고유 주소를 할당할 수 있습니다. 주소는 일반적으로 4개의 옥탯(octets)으로 나뉘어 십진수로 표현됩니다(예: 192.0.2.1).

- IPv6(Internet Protocol version 6): 인터넷의 급격한 성장과 IPv4 주소의 고갈에 따라, IPv6이 소개되었습니다. IPv6 주소는 128 비트의 길이로, 이용가능한 주소의 수가 크게 확장되었습니다. 이는 16진수 형식으로, 콜론(colon)을 통해 나눠집니다(예: 2001:0db8:85a3:0000:0000:8a2e:0370:7334). 이 표준은 이용 가능한 주소의 제한을 해결할 뿐만 아니라 라우터의 처리 간소화 및 보안 기능 향상 등 여러 측면에서 개선합니다.

두가지 IP 주소 표준은 현재 인터넷 환경에서 필수적입니다. IPv4가 현재 주로 사용되지만, 인터넷이 연결된 디바이스의 급증에 따라 인터넷 주소가 계속해서 증가하여 IPv6로의 전환이 점진적으로 이루어지고 있습니다.

## 4.4 도메인 이름

도메인 이름은 인터넷에서 웹사이트를 직관적이고 사용자 친화적으로 식별하는 역할을 하며, 기술적인 인터넷 프로토콜(IP) 주소를 쉽게 기억할 수 있는 이름으로 변환합니다. 본질적으로, 도메인 이름은 웹 탐색의 핵심 요소로 웹 사이트를 찾고 접속하는 과정을 단순하게 해줍니다.

예를 들어, example.com과 같은 도메인 이름은 숫자 형태의 IP 주소보다 훨씬 인식하고 기억하는데 쉽습니다. 이 사용자 친화적인 시스템 덕분에 인터넷 사용자들은 복잡한 문자열(IP 주소)을 기억할 필ㄹ요없이 사이트를 찾고 방문할 수 있습니다. 각각의 도메인 이름은 고유하기 때문에, 모든 웹사이트는 자신만의 구별되는 웹 주소를 가질 수 있습니다.

도메인 이름의 구조는 계층적이며 일반적으로 '.com', '.org', '.net'과 같은 최상위 도메인(TLD)과 웹사이트 소유자가 선택한 2차 도메인으로 구성됩니다. 이러한 요소들의 결합을 통해 구체적인 IP 주소를 나타내는 도메인 이름이 만들어집니다.

<!-- TLD(Top-Level Domain, 최상위 도메인): 도메인 이름의 계층적 구조에서 최상위. 도메인의 유형이나 목적을 나타냄 -->

도메인 이름은 웹사이트의 접근을 쉽게 해줄 뿐만 아니라 기업과 개인들이 브랜드를 구축하고 온라인 정체성을 설립하는데 중요한 역할을 합니다. 디지털 시대에서, 도메인 이름은 단순한 주소를 넘어 온라인 존재감과 디지털 브랜드 전략의 핵심요소입니다.

## 4.5 DNS (도메인 네임 시스템)

도메인 네임 시스템(DNS)는 인터넷의 전화번호부와 같습니다. 사용자 친화적인 도메인 이름(예: www.example.com)을 컴퓨터가 사용하는 IP 주소로 변환합니다. DNS는 사용자 친화적인 인터넷 탐색을 위해 중요하며, 사용자들이 복잡한 숫자형 IP 주소를 기억할 필요 없이 웹사이트에 접근할 수 있도록 돕습니다.

더 알아보기:

- [DNS가 작동하는 방식(영어)](https://howdns.works/)

## 4.6 URLs (통합 자원 위치)

URL(Uniform Resource Locator)은 인터넷에서 자원을 접근하기 위해 사용되는 주소입니다. URL은 서버에서 자원의 위치와 접근하는데 사용되는 프로토콜을 지정합니다. 일반적으로 URL은 프로토콜(예: HTTP 또는 HTTPS), 도메인 이름, 자원에 대한 경로를 포함합니다.

더 알아보기:

- [URL이란?](https://developer.mozilla.org/ko/docs/Learn_web_development/Howto/Web_mechanics/What_is_a_URL) on MDN

## 4.7 서버 및 웹 호스팅

디지털 세계에서의 강력한 시스템인 서버는 인터넷과 로컬 네트워크를 통해 요청을 처리하고 데이터를 분배하는 역할을 합니다. 이러한 강력한 기계는 디지털 생태계의 핵심이며, 웹사이트 호스팅부터 복잡한 애플리케이션의 실행까지 모든 작업을 지원합니다.

웹 호스팅은 온라인 분야에서 중요한 서비스로, 서버 인프라와 안정적인 인터넷 연결을 관리합니다. 웹사이트와 온라인 애플리케이션이 방해받지 않고 운영하는데 필수적임려 다양한 운영 요구사항과 규모에 맞춘 다양한 솔루션을 제공합니다. 개인 블로그이든 거대 기업의 웹사이트든, 웹 호스팅 옵션은 각각의 요구사항과 목표에 완벽하게 맞춘 서비스를 제공합니다.

- 공유호스팅: 단일 서버가 제공하는 자원을 여러 클라이언트가 공유하는 경제적인 선택안입니다. 작은 웹사이트나 블로그에 적합하며, 비용은 저렴하지만 제한된 자원과 제어를 제공합니다.
- VPS(가상 사설 서버) 호스팅: 경제적 합리성과 기능성의 균형을 이룹니다. 클라이언트들은 서버를 공유하지만 개별 가상 환경 안에서 향상된 자원과 커스터마이징 가능성을 제공합니다.
- 전용 서버 호스팅: 클라이언트에게 전용 서버를 제공하여 최대 자원 사용 가능, 최상의 성능 및 보안을 보장합니다. 많은 트래픽을 가진 거대 기업과 웹사이트에 이상적입니다.
- 클라우드 호스팅: 클라우드에 있는 가상 서버 네트워크를 활용한 다목적이고 확장 가능한 솔루션입니다. 트래픽의 변동에 맞춰 자원을 확장할 수 있으며 동적인 트래픽 패턴을 가진 사업에 적합합니다.

웹 호스팅 솔루션을 선택하는 데는 사업 규모, 예산 제약, 트래픽 수준 및 특정 기술 요구사항 등의 여러 요인이 영향을 미칩니다. 서버 호스팅 기술의 지속적인 발전과 다양화는 모든 규모의 사업들이 온라인 존재감을 구축하고 향상시키는데 효과적입니다.

## 4.8 CDN (콘텐츠 전송 네트워크)

콘텐츠 전송 네트워크(CDN, Content Delivery Network)는 콘텐츠 배포 기술에서 주요한 발전을 나타냅니다. CDN은 다양한 지리적 위치에 전략적으로 분포된 광범위한 네트워크입니다. 이 네트워크는 전 세계 사용자에게 인터넷 콘텐츠를 빠르고 효율적으로 전달합니다.

CDN은 사용자의 위치에 더 가까운 서버에 웹 페이지, 이미지 및 비디오 스트림 등의 콘텐츠를 캐시함으로써 지연 시간을 크게 줄입니다. 이 설정은 특히 높은 트래픽을 처라하는 웹사이트와 전세계자의 사용자를 기반으로하는 온라인 서비스에 도움이 됩니다. 사용자와 가까운 서버는 빠른 접근 속도를 제공하며 로딩 시간을 줄이고 웹사이트 성능을 개선하여 전반적인 사용자 경험을 향상시킵니다.

속도 개선을 넘어서 CDN는 부하를 분산시키고 대량 트래픽을 처리하는데 기여하여 웹사이트와 웹 서비스의 신뢰성과 가용성을 증가시킵니다. 또한 트래픽 급증을 효과적으로 관리하고 잠재적인 병목 현상을 완화하여 트래픽이 몰리는 시간 대에도 일관된 콘텐츠 전달을 보장합니다.

속도와 신뢰성이 중요한 오늘날의 디지털 환경에서는, CDN의 사용이 기업들이 온라인 존재감을 최적화하고 최고의 사용자 경험을 제공하기 위한 필수적인 부분이 되었습니다.

## 4.9 HTTP/HTTPS (하이퍼텍스트 전송 프로토콜/보안)

HTTP(HyperText Transfer Protocol)와 HTTPS(HTTP Secure)은 인터넷에서 정보를 전송하는데 사용되는 기본적인 프로토콜입니다. HTTP는 월드 와이드 웹에서 데이터 통신의 기초를 형성하며, HTTPS는 이러한 통신에 보안계층을 추가합니다.

HTTP와 HTTPS의 주요 특징

- 기본 기능: HTTP는 웹 브라우저와 서버 사이의 소통을 하기 위해 고안되었습니다. 브라우저가 데이터를 요청하고 서버는 요청된 정보를 응답하는 요청-응답 구조를 다릅니다.
- HTTPS의 보안: HTTPS는 암호화가 적용된 HTTP입니다. 브라우저와 서버 간 전송되는 데이터를 암호화하여 보안을 강화하고 민감한 정보를 가로채거나 변경하는 것으로부터 보호합니다.
- 포트 번호: 기본적으로 HTTP는 80번 포트를 사용하고 HTTPS는 443번 포트를 사용합니다. 이 포트들은 웹 서버가 웹 클라이언트로부터 들어오는 연결을 듣는데 사용됩니다.
- URL 구조: URL에서 HTTP는 'http://'로, HTTPS는 'https://'로 나타납니다. 이러한 작은 차이는 웹사이트와의 연결이 암호화를 통해 보호되는지 여부를 나타냅니다.

<!-- 탬퍼링(tampering): 데이터나 시스템의 내용을 무단으로 변경하거나 수정하는 행위를 의미-->

차이점과 사용법:

- 데이터 보안: 가장 중요한 차이점은 보안입니다. HTTPS는 보안 채널을 제공하며 특히 은행, 쇼핑 및 개인 정보와 같은 민감한 데이터를 다루는 웹사이트에 중요합니다.
- SEO와 신뢰: 구글과 같은 검색엔진은 HTTPS 웹사이트를 선호하며 더 안전한 페이지로 여깁니다. 또한, 웹 브라우저는 사용자의 신뢰에 영향을 미칠 수 있어 HTTP 사이트에 대해 종종 보안 경고를 표시합니다.
- 인증서 요구사항: HTTPS를 구현하기 위해서 웹사이트는 인증된 인증 기관(CA, Certificate Authority)으로부터 SSL/TLS 증명서를 발급받아야 합니다. 이 인증서는 신뢰할 수 있고 암호화된 연결을 구축하는데 중요합니다.
- 성능: 암호화 과정 때문에 HTTP보다 HTTPS가 더 느렸지만, 현재에는 기술 발전 덕분에 이 성능 차이가 크게 줄어들었습니다.

HTTP와 HTTPS의 차이점을 이해하는 것은 웹 개발자와 사용자에게 중요합니다. 그 선택은 웹사이트의 보안, 사용자 신뢰 및 검색 엔진 순위에 상당한 영향을 미칠 수 있습니다.

## 4.10 웹 브라우저

웹 브라우저는 복잡한 소프트웨어 애플리케이션으로 월드 와이드 웹에 접속하고 상호작용하는데 중요한 역할을 합니다. 사용자와 웹 콘텐츠 사이의 인터페이스 역할을 하며 웹 페이지를 렌더링하고 원활한 사용자 경험을 제공합니다. 웹 브라우저의 기능과 특징을 더 깊이 알아보겠습니다:

웹 브라우저의 주요 기능:

- 웹 콘텐츠 렌더링: 브라우저는 HTML, CSS 및 자바스크립트 파일애 작성된 콘텐츠를 번역하고 전시합니다. 구조화하기 위해서는 HTML을, 표현을 위해서는 CSS를, 상호작용을 위해서는 자바스크립트를 처리하며 그들을 시각적이고 상호작용이 가능한 웹사이트로 변경합니다.
- 요청과 응답 순환: 사용자가 웹페이지를 요청할 때, 브라우저는 서버에게 페이지가 어디에 호스트되었는지 요청을 보냅니다. 서버는 필수적인 파일(HTML, CSS, 자바스크립트, 이미지 등)을 응답하며 브라우저는 이를 통해 페이지를 렌더합니다.
- 자바스크립트 실행: 현대 브라우저는 폼 검증, 애니메이션, 비동기 데이터 요청 등과 같은 웹 페이지의 동적인 상호작용을 할 수 있도록 자바스크립트 코드를 실행하는 자바스크립트 엔진을 내장하고 있습니다.

브라우저 작동의 비하인드:

- 파싱: 브라우저는 웹페이지의 구조, 스타일 및 동작을 이해하기 위해 HTML, CSS 및 자바스크립트 파일을 파싱합니다.
- 렌더링 엔진: 각 브라우저는 사용자가 화면에서 볼 수 있게 웹 콘텐츠를 변환하는 렌더링 엔진을 가지고 있습니다. 이는 레이아웃 계산, 스타일 계산 및 최종 시각적 결과물 그리기를 수행합니다.
- 최적화: 현대 브라우저는 캐싱(자주 접근하는 자원의 복사본을 저장)과 느린 로딩(중요하지 않은 자원을 필요할때만 로딩)과 같은 기술을 통해 성능을 최적화합니다.

웹 개발에서 브라우저의 역할:

- 크로스 브라우저 호환성: 개발자는 각 고유한 특성과 렌더링 동작을 가진 다양한 브라우저에서 웹사이트 기능의 동작을 보장해야합니다.
- 접근성: 장애를 가진 사람들을 포함한 모든 사용자들이 웹 콘텐츠에 접근할 수 있도록 돕는 기능을 제공해야합니다.

## 4.11 자바스크립트 엔진

때로는 자바스크립트 가상 머신이라고 불리는 자바스크립트 엔진은 자바스크립트 코드를 처리, 컴파일, 실행하는 특별한 소프트웨어 구성 요소입니다. 자바스크립트는 고수준의 인터프리터형 스크립트여서 컴퓨터가 이해할 수 있는 실행 가능한 코드로 변환할 수 있는 엔진이 필요합니다. 이러한 엔진은 웹 브라우저에서 뿐만 아니라 서버와 같은 다른 환경에서도 사용됩니다(Node.js는 V8 엔진을 사용).

자바스크립트 엔진의 주요 기능:

- 파싱: 엔진은 자바스크립트 코드를 읽고 이를 이해할 수 있는 요소(토큰)로 나눈 후, 프로그램의 문법 구조를 나타내는 추상 구문 트리(Abstract Syntax Tree, AST)를 구성합니다.
- 컴파일: 현대의 자바스크립트 엔진은 즉시 컴파일(Just-In-Time, JIT) 기법을 사용합니다. 이 과정은 많은 엔진에서 두 단계로 나뉩니다.

  - 기본 컴파일: 자바스크립트를 더 작은 중간 코드로 빠르게 변환합니다.
  - 최적화 컴파일: 더 최적화된 기계어 코드로 더 컴파일하여 성능을 향상시킵니다. 엔진은 특정한 가정이 유효하지 않으면 코드를 최적화하지 않을 수도 있습니다.

- 실행: 컴파일된 코드는 컴퓨터의 프로세서에서 실행됩니다.
- 최적화: 실행 중 엔진은 실시간으로 코드 성능을 최적화하기 위해 데이터를 수집하고 종종 더 좋은 효율성을 위해 코드를 다시 컴파일합니다.

주요 자바스크립트 엔진:

- V8(Google Chrome, Node.js, Microsoft Edge): 속도와 효율성으로 잘 알려져 있으며 자바스크립트를 실행하기 전에 이를 네이티브 기계어로 직접 컴파일합니다.
- SpiderMonky(Mozilla Firefox): 최초의 자바스크립트 엔진이었으며 성능과 확장성에 중점을 두고 크게 발전했습니다.
- JavaScriptCore(Safari): Nitro로도 알려져 있으며 실행 효율성을 강조합니다.
